Running the Messaging Infrastructure Tests

This page outlines how to set up and run the messaging infrastructure tests. It assumes the messaging systems to be tested have been installed and the queues set up.

Setting Up the Message Listeners

The message listener is a small piece of code which listens for messages on the request queue and put replies onto the reply queue, as defined in the reply to queue parameter of the message.

The message listener can return different results based on the test run from the command line or the GUI.

The message listener supports two types of results: echo, and reply.
Echo
Echo returns the message sent back onto the reply queue.
Reply
This allows the testing of reply messages of various sizes and attempts to mimic a real system which could return messages of various sizes and types.

If the message is an instance of javax.jms.TextMessage and it is in the following format:

<?xml version="1.0"?>
<DataRequest>
  <ReplyType>text</ReplyType>
  <ReplySize>1024</ReplySize>
</DataRequest>

the message listener will create a random string of 1024 bytes in length and put it onto the reply queue as a javax.jms.TextMessage. If the <ReplyType> element has the word binary in it, the message listener will create a javax.jms.BytesMessage instead.

If the text message is not XML or not in the above format, the message listener will put a message like the one below onto the reply queue:

TextMessage processed at: Fri Aug 05 04:45:34 NZST 2005

If a binary message is put onto the reply queue, the message listener will put a message like the one below onto the reply queue:

BytesMessage processed at: Fri Aug 05 04:45:34 NZST 2005

Building the Messaging Infrastructure details how to checkout the code and the Ant targets.

The all_messagelisteners Ant target creates the message listening code and scripts for all the supported messaging systems in the MSGINF_HOME/build/env directory, one directory for each messaging system.

Beacuse the MSGINF_HOME/build directory could be cleaned out, the message listening code and scripts created in the env directory need to be moved to another directory, e.g. C:\messagelisteners.

This will create the following directory and file structure in the C:\messagelisteners directory:

Running the Tests From the Command Line

When a test is run for a particular messaging system, the messaging system server must be running first and the corresponding message listener must also be running in a DIFFERENT DOS or Unix session.

The test Ant target creates the test script MSGINF_HOME/build/test/test.sh:

All the scripts take the same required parameters:

Parameter Name Example Value Description
Test Name
 submit|echo|reply submit: puts the message onto the submit queue defined in the connector below.
echo: puts the message onto the request queue defined in the connector below and tells the message listener to put an echo message onto the reply queue.
reply: puts the message onto the request queue defined in the connector below and tells the message listener to put a reply message onto the reply queue of the type and size defined in the XML data in the message.
System
 activemq
 The name of the system to be used as defined in the xml properties file.
Connector Name
 activemq_rr_text_consumer
 The name of the connector to be used as defined in the xml properties file.
Number of Threads 1
 The number of threads to use in the test.
Number of Messages in Thread 10
 The number of messages to send per thread.
Data file ../../data/request.xml Relative path from the MSGINF_HOME/build/test directory to a valid file. The data in the file will be sent in each message.
Port 9955 The port the shutdown thread is listening to. If a process connects to this port, the messaging infrastructure will shut down immediately.

The following are examples of tests:

Scenario Command Line Command
Send 1000 XML requests to ActiveMQ in 10 threads using the consumer message requester while expecting an echoed reply.
 test.sh echo activemq activemq_rr_text_consumer 10 100 ../../data/request.xml 9955
Send 1000 XML requests to ActiveMQ in 10 threads using the text consumer message requester while expecting a binary reply of 100K.
 test.sh reply activemq activemq_rr_text_consumer 10 100 ../../data/request_binary_100K.xml 9955
Send 1000 binary files to ActiveMQ in 10 threads while NOT expecting a reply. test.sh submit activemq activemq_submit_binary 10 100 ../../data/8520.pdf 9955

Running the Tests Using the GUI Application

Instead of running the tests using the command line, a GUI application can be used instead.

This GUI application asks the user for the parameters to the test script above and runs the test. First the XML properties file needs to be loaded to give the user a list of available messaging systems and connectors.
